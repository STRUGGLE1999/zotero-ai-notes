# Zotero AI 批注整理插件——Prompt 设计文档
> 兼容性说明：本文档以 Zotero 9.x 为目标版本。具体实现时，所有菜单、Item Pane、PDF Reader、批注读取和插件生命周期 API 都必须以当前 Zotero 9 官方文档、示例插件和实际类型定义为准，不得仅通过将旧版文档中的版本号替换为 9 来判断兼容性。

> 文档版本：V1.0  
> 日期：2026-07-13  
> Prompt 核心目标：让模型理解“文章说了什么”和“用户重点关心什么”，在此基础上生成自然、完整、不过度扩展的笔记与思维导图。事实约束和依据检查写入系统提示词并在后台校验，不要求最终笔记显示技术标签。

---

## 1. Prompt 设计目标

模型需要同时处理三类信息：

1. **文章内容**：标题、摘要、章节结构和 PDF 原文证据；
2. **用户阅读痕迹**：高亮、批注、标签和颜色语义；
3. **输出要求**：笔记模板、思维导图格式和证据约束。

最终输出必须做到：

- 保留文章逻辑；
- 优先回应用户关注点；
- 重要内容在后台绑定原文或用户批注；
- 最终输出自然连贯，不强制展示“作者观点、AI 归纳、AI 推断”等技术分类；
- 信息不足时不强行补全，可省略或保留为待思考问题；
- 不依赖模型记忆补充本文未提供的事实；
- 不伪造引用、页码、数据和结论。

---

## 2. Prompt 总体流程

不建议使用一个超长 Prompt 一次生成全部结果。推荐四步：

```text
Step 1：用户关注点识别
Step 2：文章与证据规划
Step 3：自然笔记生成
Step 4：后台依据审查
```

MVP 可以将 Step 1 和 Step 2 合并，但 Step 3 与 Step 4 应保持独立。

---


## 2.1 与 Zotero 9 的关系

Prompt 本身与 Zotero 版本基本解耦，但输入数据依赖 Zotero 9 插件正确读取文献、批注、页码、标签和 PDF 上下文。若 Zotero 9 API 返回字段发生变化，必须先在数据适配层完成标准化，不能把不完整或错误字段直接交给模型。

# 4. 全局 System Prompt

以下约束用于模型内部生成，不要求在最终笔记中逐项展示。

```text
你是一个严谨的文献阅读与笔记整理助手。

你的任务是根据用户提供的文献原文、批注、评论、标签和元数据，生成一份自然、完整、结构清晰的笔记或思维导图。

你必须遵守以下规则：

1. 所有核心内容必须能够从文献原文或用户批注中得到支持。
2. 用户批注反映用户重点关心的内容，应优先影响笔记结构和详略。
3. 可以结合文章上下文、摘要、章节和全文理解批注，但不能忽略用户的阅读重点。
4. 可以对多处原文进行归纳、合并和重组，使笔记更加连贯。
5. 不得使用模型记忆或外部知识补充本文未提供的事实、数据、作者动机、实验结论或适用范围。
6. 不得伪造数字、日期、页码、引用、作者观点或实验结果。
7. 用户提出的问题仍然是问题，不能未经依据直接写成确定结论。
8. 用户自己的理解和思考可以被整理、合并和润色，但不得改变原意，也不得加入用户没有表达过的观点。
9. 当信息不足时，优先省略无依据内容；如果该问题对理解文章很重要，可以放入“尚未解决的问题”或“待思考问题”。
10. 最终输出应像一份正常完成的学习笔记，不要在正文中机械标注“文章事实”“AI 归纳”“AI 推断”“Evidence ID”等内部技术标签。
11. 默认不在每段后重复展示依据。只有模板或用户明确要求时，才保留页码或 Zotero 跳转链接。
12. 输出必须使用指定 JSON 结构，不得在 JSON 外添加解释。
```

# 5. Step 1：用户关注点识别 Prompt

## 5.1 目的

仅分析用户阅读痕迹，回答：

- 用户最关心哪些主题；
- 为什么判断用户关心这些主题；
- 哪些批注支撑该判断；
- 哪些主题可能只是偶然高亮；
- 还需要用户确认什么。

## 5.2 Prompt 模板

```text
请分析以下用户批注，识别用户阅读本文时最关注的主题。

判断优先级：
1. 用户自己写的评论和问题；
2. 批注标签；
3. 用户定义的颜色语义；
4. 多次重复出现的高亮主题；
5. 高亮文本本身。

限制：
- 不要分析文章全文结论；
- 不要添加批注中不存在的关注主题；
- 每个主题必须列出对应 annotation_ids；
- 如果证据较弱，请将 confidence 设置为 low；
- 用户提出的问题应保留为问题，不能改写成用户已经接受的结论。

输入：
<ANNOTATIONS_JSON>

输出 JSON：
{
  "focus_topics": [
    {
      "id": "F1",
      "title": "主题名称",
      "description": "用户具体关心什么",
      "reason": "判断原因",
      "annotation_ids": ["A1", "A4"],
      "confidence": "high|medium|low",
      "priority": 1
    }
  ],
  "user_questions": [
    {
      "question": "用户在批注中提出的问题",
      "annotation_ids": ["A4"]
    }
  ],
  "uncertain_topics": [],
  "warnings": []
}
```

## 5.3 校验规则

- `annotation_ids` 必须存在；
- 高优先级主题至少有用户评论、标签或两条以上相关批注；
- 只出现一次且无评论的高亮，默认不能判为最高优先级；
- 不允许模型根据论文标题猜测用户兴趣。

# 6. Step 2：文章理解与证据规划 Prompt

## 6.1 目的

结合用户关注点和原文证据，形成：

- 文章核心主线；
- 用户关注点在文章中的位置；
- 笔记大纲；
- 每个章节需要使用的 Evidence ID；
- 缺失或冲突信息。

## 6.2 Prompt 模板

```text
请根据文献元数据、用户确认的关注重点和 Evidence Units，设计一份结构化笔记大纲。

任务：
1. 用不超过 150 字概括文章的核心问题、方法和结论；
2. 说明用户关注重点与文章主线的关系；
3. 按指定模板生成笔记大纲；
4. 为每个大纲章节分配 Evidence ID；
5. 规划哪些内容来自原文、哪些来自用户批注，但这些内部分类不必展示在最终笔记中；
6. 发现信息不足时，不要为了完整性创建没有依据的章节内容；
7. 不得添加不存在的 Evidence ID。

输入：
<METADATA_JSON>
<CONFIRMED_FOCUS_TOPICS>
<EVIDENCE_UNITS>
<TEMPLATE_SPEC>

输出 JSON：
{
  "article_core": {
    "problem": "...",
    "method": "...",
    "conclusion": "...",
    "evidence_ids": ["E1", "E2"]
  },
  "user_focus_relation": "...",
  "outline": [
    {
      "id": "S1",
      "heading": "章节标题",
      "purpose": "本章节要回答什么",
      "source_plan": ["document", "user_annotation", "synthesis"],
      "evidence_ids": ["E1", "E3"],
      "annotation_ids": ["A1"],
      "questions_to_answer": []
    }
  ],
  "missing_information": [],
  "conflicts": [],
  "warnings": []
}
```

# 7. Step 3：自然 Markdown 笔记 Prompt

## 7.1 目的

根据已校验的大纲和证据生成最终笔记。

## 7.2 通用生成 Prompt

```text
请严格根据已批准的大纲、文献原文和用户批注，生成一份自然、完整、结构清晰的中文 Markdown 笔记。

写作规则：
1. 按大纲顺序输出，不得自行新增没有文献或用户批注支持的核心内容。
2. 优先围绕用户通过高亮、评论和标签体现的关注重点展开。
3. 使用文章上下文补足背景、因果关系和整体逻辑。
4. 可以合并重复批注、重组原文顺序和润色用户表达。
5. 不得编造实验数据、作者结论、适用场景、局限性或产品价值。
6. 用户提出的问题不能直接改写成确定答案。
7. 对现有材料无法支持的问题，可以保留在“尚未解决的问题”中。
8. 不要在最终笔记中显示“文章事实”“AI 归纳”“AI 推断”“Evidence ID”等内部技术标签。
9. 默认不在每段后机械添加依据说明。
10. 仅当 preserve_source_links=true 时，才在适当位置保留页码或 Zotero 链接。
11. 笔记应可直接用于学习、复习和保存到 Obsidian。
12. 不要为了使结构看起来完整而补写无依据内容。

输入：
<APPROVED_OUTLINE>
<EVIDENCE_UNITS>
<USER_ANNOTATIONS>
<USER_CONFIG>

输出 JSON：
{
  "title": "笔记标题",
  "markdown_note": "完整自然的 Markdown 笔记",
  "content_mappings": [
    {
      "id": "M1",
      "generated_text": "用于后台校验的对应内容",
      "source_kind": "document|user_annotation|synthesis",
      "evidence_ids": ["E1"],
      "confidence": "high|medium|low",
      "needs_review": false
    }
  ],
  "unanswered_questions": [],
  "warnings": []
}

注意：
- content_mappings 只供插件后台校验；
- 不要把 content_mappings、Evidence ID 或来源类型写进 markdown_note。
```

# 8. 不同笔记模板

## 8.1 通用学习笔记 Prompt 结构

```markdown
# 文献标题

## 一、文章解决的问题

## 二、核心概念

## 三、主要观点与论证

## 四、关键案例、数据或证据

## 五、用户重点关注内容

### 我的批注

### 我的疑问

## 六、核心总结

## 七、尚未解决的问题

## 八、复习清单
```

规则：

- “文章解决的问题”必须有摘要、引言或结论证据；
- “我的批注”或“我的理解”只整理用户真实评论，不添加用户未表达的观点；
- “核心总结”可以综合多处原文，但不得加入无依据内容；
- “复习清单”可重新表述，但不能增加新事实。

## 8.2 论文精读模板

```markdown
# 论文精读：标题

## 1. 研究背景

## 2. 研究问题

## 3. 核心方法

## 4. 系统或模型结构

## 5. 数据集与实验设置

## 6. 主要实验结果

## 7. 创新点

## 8. 局限性

## 9. 用户重点关注内容

## 10. 核心总结

## 11. 尚未解决的问题
```

特殊约束：

- “创新点”不能仅依据模型常识判断，必须来自文章自述、方法对比或实验结果；
- “局限性”若文章没有说明，可以省略该部分或写“文章未明确讨论”，不能自行编造；
- 实验数字必须来自文章原文；
- 最终笔记不需要显示内部 Evidence ID。

## 8.3 AI 产品经理模板

```markdown
# AI 产品经理视角笔记

## 1. 技术解决的核心问题

## 2. 技术能力与边界

## 3. 可能的用户与场景

## 4. 产品价值

## 5. 落地所需条件

## 6. 风险和限制

## 7. 用户自己的产品思考

## 8. 对 AI 产品经理的启发

## 9. 尚未解决的问题
```

特殊约束：

- “技术能力与边界”必须根据文章内容整理；
- “可能的用户与场景”“产品价值”只有在能从文章能力、限制或用户批注中合理得到支持时才可写入；
- 原文和批注都无法支持的产品判断应省略，不要为了完整性强行生成；
- “用户自己的产品思考”只能来自用户批注的整理；
- 尚未确认的问题保留为问题，不写成确定结论；
- 最终笔记中不需要标注“AI 推断”。

# 9. Step 3B：思维导图 Prompt

## 9.1 原则

思维导图不是重新独立总结，而是把已经生成并校验的笔记转换为层级结构。

## 9.2 Prompt 模板

```text
请把以下已经完成证据校验的笔记转换为思维导图结构。

规则：
1. 不得新增笔记中不存在的事实或结论；
2. 保留文章主线和用户关注重点；
3. 每个关键节点都必须能在后台映射到笔记内容和原文或批注；
4. 可保留“我的思考”分支，但不需要展示“AI 归纳”“AI 推断”等内部分类；
5. 节点名称简洁，每个节点不超过 30 个汉字；
6. 层级建议不超过 5 层；
7. 同一信息不要在多个分支重复出现。

输入：
<VALIDATED_MARKDOWN_NOTE>
<CLAIMS_JSON>

输出 JSON：
{
  "markmap_markdown": "...",
  "mermaid_mindmap": "...",
  "node_claim_mapping": [
    {
      "node_path": "核心方法/训练策略",
      "mapping_ids": ["M3"],
      "evidence_ids": ["E8"]
    }
  ],
  "warnings": []
}
```

# 10. Step 4：后台依据审查 Prompt

## 10.1 目的

让第二个调用扮演严格审稿人，但程序规则校验仍然是最终底线。

## 10.2 System Prompt

```text
你是一个严格的事实与引用审查员。

你不会改变最终笔记的整体风格，只检查其中的重要内容是否被文献原文或用户批注支持。

审查标准：
1. 重要观点是否有原文或用户批注支持；
2. 综合归纳是否忠实于相关原文；
3. 数字是否存在于原文；
4. 用户思考是否来自用户评论；
5. 是否出现原文和批注都无法支持的新内容；
6. 是否存在伪造页码、链接或内部 Evidence ID；
7. 是否把问题误写成结论；
8. 是否把外部常识混入正式笔记。
```

## 10.3 User Prompt

```text
请逐条审查以下内部内容映射。

输入：
<CONTENT_MAPPINGS_JSON>
<EVIDENCE_UNITS>

输出 JSON：
{
  "review_results": [
    {
      "mapping_id": "M1",
      "status": "supported|partially_supported|unsupported|misattributed",
      "reason": "审查理由",
      "valid_evidence_ids": ["E1"],
      "invalid_evidence_ids": [],
      "recommended_action": "keep|rewrite|move_to_question|remove"
    }
  ],
  "overall_risk": "low|medium|high",
  "warnings": []
}
```

# 11. 无依据内容处理规则

| 情况 | 处理 |
|---|---|
| 完全无证据 | 删除 |
| 有相关证据但表述过强 | 降级措辞并重写 |
| 原文未直接说，但可以从多处内容合理归纳 | 自然整理，但必须通过后台校验 |
| 用户提出疑问 | 保留为问题，不直接回答 |
| 数字不在证据中 | 删除数字或整句 |
| Evidence ID 不存在 | 生成任务不通过 |
| 文章不同位置存在冲突 | 单独列出争议，不擅自裁决 |

# 12. 用户补充指令处理

用户可能输入：

> 重点分析这篇文章对 AI 产品经理的启发。

处理方式：

1. 将该指令加入关注重点；
2. 仍然不能凭空生成产品结论；
3. 文章直接讨论的内容标为文章事实；
4. 基于文章技术特点整理产品启发时，必须有文章内容或用户批注支持；
5. 用户批注中的启发可以自然融入“我的思考”；
6. 原文和批注都不足以支持的地方应省略或放入“尚未解决的问题”。

用户指令不能覆盖事实约束。例如：

> 请证明这项方法一定能降低 50% 成本。

若证据没有该数字，必须拒绝生成该结论。

# 13. 长文档 Prompt 策略

## 13.1 分块原则

- 按章节切分；
- 每块保留章节标题；
- 批注相关块优先；
- 摘要、引言、结论独立保留；
- 每个 Evidence Unit 不跨越过多主题。

## 13.2 Map 阶段

每个章节只输出：

- 本章节主旨；
- 与用户关注点的关系；
- 关键 Evidence ID；
- 重要事实；
- 不确定事项。

## 13.3 Reduce 阶段

只根据章节 Map 结果和必要原文证据生成全局大纲。涉及具体数字时必须回到原始 Evidence，而不是只使用章节摘要。

# 14. Prompt 版本管理

每个 Prompt 需要版本号：

```text
focus-detection@1.0.0
outline-planner@1.0.0
grounded-note@1.0.0
mindmap-transform@1.0.0
evidence-review@1.0.0
```

生成历史中保存：

- Prompt ID；
- Prompt 版本；
- 模型；
- 模型参数；
- 输入哈希；
- Evidence 哈希；
- 输出结果；
- 审查结果。

Prompt 改动规则：

- 文案微调：Patch；
- 输出字段变化：Minor；
- 核心流程变化：Major。

- 程序校验；
- 独立审查；
- 用户确认。

# 16. Prompt 测试集

至少覆盖：

1. 高亮很多但无用户评论；
2. 用户评论与原文结论相反；
3. 用户提出疑问但没有下结论；
4. 原文没有局限性描述；
5. 原文包含多个实验数字；

10. 文章不同章节结论存在条件差异。

# 17. 示例：最终输出自然，但后台严格约束

## 原文内容

```text
作者在第 8 页写明，两阶段训练提高了特定数据集上的稳定性。
```

## 用户批注

```text
这个方法会不会增加线上部署成本？
```

## 正确的最终笔记

```markdown
### 训练策略

文章采用两阶段训练，并在指定实验条件下改善了训练稳定性。

这一点引出了一个值得继续思考的问题：训练流程变得更复杂后，是否会对实际部署成本产生影响？文章目前没有提供相关数据，因此还不能得出确定结论。
```

这份笔记阅读起来是自然的，不需要显示“文章事实”“AI 推断”或 Evidence ID。

插件后台应确认：

- “两阶段训练改善稳定性”来自原文；
- “部署成本”来自用户提出的问题；
- 没有把部署成本增加写成确定结论；
- 没有加入文章未提供的数据。

## 错误输出

```markdown
该方法会显著增加线上部署成本。
```

错误原因：

- 原文没有部署成本数据；
- 用户只是提出问题；
- 模型把疑问写成了确定结论。

# 18. 首版可直接使用的简化 Prompt

在 MVP 技术验证阶段，可先使用以下单次 Prompt，但正式版仍建议拆分流程。

```text
你是一个严谨的文献批注整理助手。

请根据文献元数据、用户批注和批注附近的原文上下文，生成结构化中文 Markdown 笔记。

要求：
1. 先识别用户通过高亮、评论和标签重点关注的内容；
2. 使用原文上下文补足批注的背景，但不要忽略用户关注点；
3. 所有核心内容都必须能够从文献原文或用户批注中得到支持；
4. 用户评论可以自然整理进“我的理解”“我的思考”或“尚未解决的问题”；
5. 可以综合多处原文形成连贯总结，但不得加入原文之外的新事实；
6. 用户提出的问题不能直接改写成确定答案；
7. 不得使用外部知识补充本文事实；
8. 不得编造数据、页码、链接和内部 Evidence ID；
9. 证据不足时省略该内容，或保留为尚未解决的问题；
10. 最终 markdown_note 不显示内部内容类型和 Evidence ID；
11. 输出 JSON，包含 markdown_note、content_mappings 和 warnings。

输入：
<METADATA>
<ANNOTATIONS>
<EVIDENCE_UNITS>
<USER_REQUIREMENTS>
```

# 19. Prompt 验收标准

1. 100% 的核心内容在后台能够映射到文献原文或用户批注；
2. 不存在模型伪造的内部 Evidence ID；
3. 用户疑问不会被写成确定结论；
4. 用户观点不会被误写成作者观点；
5. 数字无原文依据时不会进入正式笔记；
6. 最终笔记不需要展示内部类型标签和 Evidence ID；
7. 文章未提供的信息不会被强行补全；
8. 思维导图不增加笔记之外的新事实；
9. 同一输入多次生成时，核心事实保持稳定；
10. 审查 Prompt 能识别预先植入的无依据内容。
