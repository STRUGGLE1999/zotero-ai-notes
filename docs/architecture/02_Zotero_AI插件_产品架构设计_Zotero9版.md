# Zotero AI 批注整理插件——产品架构设计
> 兼容性说明：本文档以 Zotero 9.x 为目标版本。具体实现时，所有菜单、Item Pane、PDF Reader、批注读取和插件生命周期 API 都必须以当前 Zotero 9 官方文档、示例插件和实际类型定义为准，不得仅通过将旧版文档中的版本号替换为 9 来判断兼容性。

> 文档版本：V1.0  
> 日期：2026-07-13  
> 当前产品范围：仅开发 Zotero 9 桌面端插件  
> 核心目标：根据用户在 PDF 中的高亮、评论、标签等阅读痕迹识别用户关注重点，再结合 PDF 原文上下文生成自然、完整、可靠的结构化笔记或思维导图。依据映射与防幻觉校验在插件后台完成，默认不打扰最终阅读体验。

---

## 1. 架构设计目标

本插件不是一个“自动替用户读论文”的通用摘要工具，而是一个以用户真实阅读行为为中心的知识整理工具。

完整价值链：

```text
用户阅读 PDF
→ 用户高亮、批注、加标签
→ 插件识别用户关注重点
→ 插件读取相关原文和必要上下文
→ 大模型进行有依据的归纳
→ 生成结构化笔记或思维导图
→ 用户审核
→ 写回 Zotero / 导出 Markdown
```

架构必须同时保证：

1. 用户批注是生成任务的核心线索；
2. 模型可以利用文章全文或相关上下文理解批注；
3. 生成过程中的关键内容都必须能够在文献原文或用户批注中找到支撑；
4. 最终呈现给用户的是自然、连贯、完整的笔记或思维导图，不强制展示内部证据标签；
5. 不把模型常识、外部知识或未经支持的推断写入正式笔记；
6. 不覆盖用户原始批注；
7. 用户确认后才保存生成结果；
8. MVP 不依赖自建云端后端。

## 2. 核心产品原则

### 2.1 批注决定关注重点

高亮频次、用户评论、批注标签、颜色含义和所在章节，共同构成“用户关注画像”。

例如：

```text
用户高亮模型架构较多
+ 多次评论“这个模块如何落地”
+ 使用“产品启发”标签
```

插件可判断用户更关心：

- 模型结构；
- 工程落地；
- 产品化价值。

但是，插件不能仅根据批注数量武断判断。最终应由模型结合批注内容和文章结构进行归纳，并允许用户在生成前查看、修改“本次关注重点”。

### 2.2 全文用于提供事实和上下文

批注可能只包含一句话，缺少前因后果。插件需要从 PDF 中读取：

- 批注所在段落；
- 前后相邻段落；
- 当前章节标题；
- 摘要、引言和结论；
- 必要时读取整篇全文。

全文的作用是补足语义和验证结论，而不是忽略用户批注重新生成普通摘要。

### 2.3 输出自然，生成过程可校验

插件内部需要严格区分信息来源，但这种区分主要服务于生成质量控制，不要求直接展示在最终笔记中。

内部信息来源包括：

| 内部类型 | 含义 | 处理方式 |
|---|---|---|
| 文献原文 | 文章明确表达的方法、数据、观点和结论 | 可以写入正式笔记 |
| 用户批注 | 用户写下的理解、疑问、联想和重点 | 融入对应章节，保留用户真实含义 |
| 模型归纳 | 基于原文和批注进行的结构化总结 | 可以自然写入笔记，但必须有内部依据 |
| 无依据扩展 | 原文和批注都没有支持的内容 | 不得进入正式笔记 |

最终文档不需要出现“文章事实”“AI 归纳”“Evidence ID”等技术标签，也不需要在每一段后面机械展示依据。用户看到的应当是一份正常、完整、可直接阅读和复习的笔记。

插件内部仍需完成以下检查：

- 关键结论是否能在原文或批注中找到支持；
- 数字、日期、实验结果是否来自文献；
- 用户提出的问题是否被错误改写成确定结论；
- 模型是否加入了文献和批注之外的内容；
- 思维导图是否与最终笔记保持一致。

对于无法确认的内容，优先不写入正式笔记；若它是用户明确提出的重要问题，可以作为“待思考问题”保留，而不是强行给出答案。

---|---|---:|
| 文章事实 | 文章明确提出的方法、数据、结论 | 是 |
| 用户观点 | 用户写在批注中的想法、问题和判断 | 来自用户批注 |
| AI 归纳 | 基于多处原文进行的概括 | 是，需绑定证据 |
| AI 推断 | 基于原文进行的合理推演 | 必须明确标注，默认少用 |

禁止生成：

- 文章中没有出现的实验数据；
- 文章未声明的结论；
- 将用户疑问误写成作者观点；
- 将模型外部知识混入“本文内容”；
- 没有证据的产品价值或行业判断。

## 4. 核心模块

## 4.1 Zotero UI 层

负责用户交互，不承担业务推理。

### 首版入口

- 文献右键菜单：`AI 整理批注`
- 文献右键菜单：`AI 生成思维导图`
- 设置页：模型、API、默认模板和隐私设置
- 生成窗口：关注重点、证据范围、输出形式
- 结果窗口：Markdown 编辑与渲染预览

### 后续入口

- Zotero 右侧 Item Pane 自定义区域
- 批量任务入口
- 多篇文献对比入口

Zotero 9 提供插件扩展机制和自定义 Item Pane 区域能力，插件应使用官方接口，而不是直接修改 Zotero 内部界面结构。

---

## 4.2 Zotero 数据适配层

负责将 Zotero 内部数据转换为插件自己的领域对象。

读取内容：

- 父文献条目；
- 标题、作者、年份、摘要、DOI；
- PDF 附件；
- 高亮文字；
- 用户评论；
- 批注颜色；
- 批注标签；
- 页码；
- 排序位置；
- 附件 Key 和 Annotation Key；
- 已有文献笔记。

禁止直接修改 `zotero.sqlite`。优先使用 Zotero JavaScript API，因为 Zotero 官方建议插件通过 API 访问数据，直接操作数据库更容易受版本变化影响。

---

## 4.3 文档理解层

文档理解层的任务不是直接生成总结，而是建立“文章内容索引”。

### 文本来源优先级

1. Zotero 已建立索引的 PDF 全文；
2. PDF.js 可提取文本；
3. 外部 PDF 文本解析库；
4. OCR，仅用于扫描型 PDF，暂不纳入 MVP。

### 文章结构识别

尽可能识别：

- 标题；
- 摘要；
- 一级和二级标题；
- 段落；
- 图表标题；
- 参考文献；
- 页码。

### 上下文切片

每条批注关联：

- 当前高亮；
- 所在段落；
- 前一段；
- 后一段；
- 所在章节标题；
- 页码；
- 证据块 ID。

示例：

```json
{
  "evidence_id": "E12",
  "page": 8,
  "section": "3.2 Training Strategy",
  "annotation_text": "原始高亮",
  "user_comment": "用户自己的批注",
  "context_before": "前文",
  "context_after": "后文",
  "zotero_link": "zotero://open-pdf/..."
}
```

---

## 4.4 用户关注点识别层

该模块先形成“关注重点草案”，再由用户确认。

### 信号来源

- 用户评论：权重最高；
- 批注标签；
- 用户自定义颜色语义；
- 高亮内容语义；
- 同一主题批注数量；
- 批注所在章节；
- 用户生成时填写的补充要求。

### 输出示例

```json
{
  "focus_topics": [
    {
      "topic": "模型训练策略",
      "reason": "相关批注 6 条，其中 2 条含用户评论",
      "annotation_ids": ["A3", "A4", "A8"]
    },
    {
      "topic": "产品落地限制",
      "reason": "用户使用“产品启发”和“风险”标签",
      "annotation_ids": ["A9", "A11"]
    }
  ]
}
```

用户可执行：

- 删除错误关注点；
- 调整优先级；
- 添加本次特别关注的问题；
- 选择“只根据批注整理”或“结合全文补充”。

---

## 4.5 内部依据与一致性层

这是插件防止“无中生有”的核心模块，但其结果默认不直接展示给用户。

插件在内部为原文片段和用户批注建立可追踪的数据映射，用于：

- 确认笔记内容是否来自文献或用户批注；
- 防止模型伪造实验结果、数字和作者结论；
- 在重新生成某一章节时准确找到相关原文；
- 在用户发现问题时支持定位来源；
- 对最终内容执行自动质量检查。

内部可以继续使用 Evidence Unit、Claim Mapping 等技术结构，但这些属于系统实现细节。默认输出的 Markdown 笔记不显示 Evidence ID，也不要求每一段附带“依据”标签。

仅在以下情况下向用户展示来源信息：

- 用户主动点击“查看原文依据”；
- 用户开启“保留页码链接”选项；
- 系统检测到内容可能缺乏支持，需要用户复核；
- 用户使用论文精读模板并希望保留引用页码。

## 4.6 AI 编排层

建议采用两阶段而不是一次性生成。

### 阶段 A：理解与规划

模型输入：

- 文献元数据；
- 用户批注；
- 相关上下文；
- 文章摘要和章节结构；
- 用户确认的关注重点。

模型输出：

- 文章核心主题；
- 用户重点关注；
- 笔记大纲；
- 每个章节准备使用的 Evidence ID；
- 信息缺口。

### 阶段 B：证据化生成

模型根据已批准的大纲生成：

- Markdown 笔记；
- 思维导图层级；
- Claim-Evidence 映射；
- 警告列表。

两阶段方案比一次性 Prompt 更容易控制幻觉，也便于用户在正式生成前检查方向。

---

## 4.7 结果校验层

结果校验在后台执行，目标是保证最终笔记自然、可靠，而不是让最终文档充满技术标记。

### 规则校验

- 模型使用的原文片段是否真实存在；
- 数字、日期、样本量和实验指标是否出现在文献中；
- 用户观点是否确实来自用户批注；
- 用户疑问是否被错误改写成结论；
- 是否出现文献和批注都没有支持的新观点；
- Markdown 是否完整；
- 思维导图是否与笔记内容一致。

### 校验不通过时

- 删除无依据内容；
- 将过强表述改为更谨慎、符合原文的表达；
- 要求模型根据原文和批注重新生成相关段落；
- 对重要但无法确认的问题保留为“待思考问题”；
- 仅在确实需要人工判断时向用户显示警告；
- 不自动保存未通过检查的结果。

## 4.8 输出层

### Zotero 笔记

写回为当前文献的子笔记，不修改原始批注。

### Markdown

最终 Markdown 应是一份整理完成的自然笔记，例如：

```markdown
# 标题

## 文献信息

## 文章核心问题

## 核心方法

## 关键观点与实验结果

## 用户重点关注内容

## 我的理解与思考

## 尚未解决的问题

## 核心结论
```

默认不展示以下内部技术内容：

- Evidence ID；
- Claim ID；
- “文章事实”“AI 归纳”等类型标签；
- 每段机械重复的依据说明；
- 模型校验过程。

可选保留页码或 Zotero 跳转链接，具体由用户在生成设置中决定。

### 思维导图

首版输出：

- Markmap Markdown；
- Mermaid mindmap。

每个关键节点可附页码或 Evidence ID。

## 5. 两种生成模式

## 5.1 严格批注模式

输入：

- 批注；
- 批注所在段落上下文；
- 文献元数据。

特点：

- 成本低；
- 最贴近用户关注点；
- 幻觉风险较低；
- 可能遗漏文章未批注的重要背景。

适合 MVP 默认模式。

## 5.2 批注引导的全文理解模式

输入：

- 用户批注；
- 摘要、目录、引言、结论；
- 与关注点相关的全文段落；
- 必要时全文分块摘要。

特点：

- 能补齐文章整体逻辑；
- 能解释批注上下文；
- Token 消耗更高；
- 必须明确区分“用户关注内容”和“全文补充内容”。

该模式是用户当前目标的完整实现，建议作为 V1.0 核心能力，但开发顺序在严格批注模式之后。

## 6. 数据流

```text
选择文献
→ 获取父条目及 PDF
→ 读取 annotations
→ 提取 PDF 文本和章节
→ 将 annotation 定位到上下文
→ 生成 Evidence Units
→ 推断用户关注重点
→ 用户确认
→ 生成内容大纲和内部依据计划
→ 生成 Note / Mindmap
→ 执行后台一致性与事实校验
→ 用户预览修改
→ 写回 Zotero / 导出
```

## 7. 隐私与安全架构

### 默认策略

- 默认只发送批注和必要上下文；
- 全文发送需用户主动开启；
- 生成前显示将发送的数据范围；
- API Key 只存储在本地；
- 日志不记录完整 PDF 文本；
- 允许使用 Ollama 等本地模型；
- 不建立插件开发者的云端中转服务。

### 敏感文献提示

用户首次启用全文模式时显示：

> PDF 全文或部分原文将发送给你配置的模型服务商。请确认文献不包含禁止外传的敏感信息。

## 9. 架构决策记录

### ADR-001：采用 Zotero 插件，不先做独立应用

原因：

- 用户阅读和批注行为发生在 Zotero；
- 可直接读取批注及文献关系；
- 减少文件导入导出步骤；
- 更容易形成完整用户闭环。

### ADR-002：不直接操作 Zotero SQLite

原因：

- 官方更推荐 JavaScript API；
- 数据库结构可能变化；
- 直接写数据库容易破坏用户文献库。

### ADR-003：内部保留依据映射，但默认不展示在最终笔记

原因：

- 便于程序校验和发现幻觉；
- 支持在需要时追踪到 Zotero 原文；
- 不破坏最终笔记的自然阅读体验；
- 将技术控制逻辑与用户输出体验分离。

### ADR-004：生成结果必须用户确认

原因：

- AI 整理可能出现遗漏或误读；
- 文献笔记属于用户知识资产；
- 不应让模型未经审核直接写入。

### ADR-005：先支持 OpenAI-compatible，再扩展其他协议

原因：

- 国内外模型兼容范围广；
- MVP 实现成本低；
- 模型适配层仍需保持独立，避免与单一服务商绑定。
