# Zotero AI Notes 项目约定

## 版本编号

项目遵循语义化版本（Semantic Versioning）：`主版本.次版本.补丁版本`。

- `0.x.y` 表示仍处于快速完善阶段，核心流程可用，但跨平台、兼容性或交互仍可能调整。
- 仅修复缺陷、兼容性或文案时递增补丁版本，例如 `0.3.7 → 0.3.8`。
- 增加一组完整的新能力或明显改变现有流程时递增次版本，例如加入 OPML/XMind 导出时使用 `0.4.0`。
- `1.0.0` 表示首个稳定版本。发布前至少应满足：macOS 与 Windows 实机回归通过、核心生成与写回流程稳定、设置和数据格式有兼容策略、没有已知 P0/P1 阻断问题、安装与升级文档完整。
- `1.x → 2.0.0` 仅用于存在明确不兼容变化的升级。
- 版本号不是小数；例如 `0.10.0` 晚于 `0.9.0`。
- 只有准备生成可安装 XPI 时才修改版本号；普通开发提交不必逐次升级版本。

## GitHub Release 策略

代码版本与 GitHub Release 分开管理，不为每一个内部测试版或普通补丁创建 Release。

默认创建 GitHub Release 的情况：

- 新的次版本或主版本，例如 `0.4.0`、`1.0.0`；
- 包含重要用户能力的里程碑版本；
- 修复无法安装、无法启动、无法生成、数据覆盖、安全或严重兼容性问题的关键补丁；
- 需要让非开发用户集中下载和测试的公开测试版本。

默认不创建 GitHub Release 的情况：

- 仅供开发者本地验证的中间版本；
- 不影响用户使用的代码整理、测试或文档小改动；
- 很快会被后续补丁合并替代的普通修复。

发布要求：

1. Release 可以合并多个连续版本的改进；例如跳过 `0.3.6`，由 `0.3.7` 一并发布其生成流程优化和关键修复。
2. 发布前必须运行 `npm run check`，并完成至少一次 Zotero 实机核心流程验证。
3. Release 标签使用 `vX.Y.Z`，安装包使用 `zotero-ai-notes-X.Y.Z.xpi`。
4. `package.json`、`package-lock.json`、`addon/manifest.json`、README、CHANGELOG 和 `updates.json` 中的公开版本必须一致。
5. `updates.json` 只指向已经公开发布的最新 XPI，并填写该文件真实的 SHA-256。
6. 已发布的同版本 XPI 不得被静默替换；二进制发生变化时必须递增版本号。
7. 在跨平台稳定版门槛达成前，`0.x` GitHub Release 默认标记为 Pre-release；`1.0.0` 起再使用正式稳定版标记。

## 建议发布节奏

- `0.3.x`：修复当前生成闭环与兼容性问题；仅关键补丁发布 Release。
- `0.4.0`：适合作为 OPML/XMind 或思维导图明显升级的里程碑 Release。
- `0.5.x`：继续完善编辑、失败恢复和生成体验。
- `0.9.0`：功能基本冻结，集中进行 macOS、Windows 和 Linux 发布回归。
- `1.0.0`：满足稳定版门槛后发布并开始承诺兼容性。
